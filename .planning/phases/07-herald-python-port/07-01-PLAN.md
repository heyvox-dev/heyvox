# Plan 07-01: Herald Python Orchestrator

## Objective

Replace `heyvox/herald/lib/orchestrator.sh` with a pure Python equivalent
(`heyvox/herald/orchestrator.py`) that eliminates all bash/Python boundary
crossings per TTS request. The Python orchestrator runs as a daemon, reads
WAV files from the queue, handles audio ducking via CoreAudio ctypes, workspace
switching, hold queue, and pause/resume - with equivalent behavior to the
bash version.

## Context

The bash orchestrator (`orchestrator.sh`) is the main playback daemon for
Herald. It polls the queue directory every 300ms for WAV files, plays them
with `afplay`, handles pause/resume via flag files, manages the hold queue for
workspace switching, and uses `osascript` for volume control. Every volume
read/write is a separate `osascript` invocation - expensive, slow, and fragile.

The goal: same behavior, pure Python, using:
- `subprocess` for `afplay` (WAV playback - keep this, it is reliable)
- `ctypes` for CoreAudio volume read/write (replaces osascript)
- `threading` for the watchdog (kill playback when recording starts)

Key files:
- `heyvox/herald/lib/orchestrator.sh` - the bash original to port
- `heyvox/herald/lib/config.sh` - constants (already have Python equivalents)
- `heyvox/herald/__init__.py` - Python Herald package

## Tasks

### Task 1: Create CoreAudio volume helper module
type: auto

Create `heyvox/herald/coreaudio.py` with:
- `get_system_volume() -> float` - reads macOS output volume (0.0-1.0) via CoreAudio ctypes
- `set_system_volume(vol: float) -> None` - sets output volume via CoreAudio ctypes
- `is_system_muted() -> bool` - checks if system output is muted
- Cached reads: `get_system_volume_cached(ttl=5.0)` returns cached value, re-reads after TTL expires
- Falls back to osascript if CoreAudio ctypes fails (defensive)
